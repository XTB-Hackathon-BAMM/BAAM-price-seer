DDD / Hexagonal Architecture Overview

The following key properties prove that BAMM-price-seer follows DDD and Hexagonal Architecture (Ports & Adapters) principles:

  • ⬢
    Framework-free domain core — The domain/ package contains only plain Kotlin data classes, enums, and interfaces. It has zero imports from Spring, Kafka, JDBC, or any other framework — making business logic portable and testable in isolation.
  • ⬢
    Explicit port interfaces define the system boundary — Four domain-defined ports (PriceRepository, PredictionPort, PredictionStrategy, SentPredictionRepository) decouple the application from any specific technology choice.
  • ⬢
    Dependency Inversion — all arrows point inward — Infrastructure adapters (JdbcPriceRepository, PredictionProducer) implement domain interfaces. The domain and application layers never reference infrastructure classes — dependencies flow exclusively from outer layers to inner layers.
  • ⬢
    Driving vs. Driven adapter separation — Inbound (driving) adapters like MarketPriceConsumer and PredictionScheduler call into the application service. Outbound (driven) adapters like PredictionProducer and JDBC repositories are injected via Spring DI and implement domain ports.
  • ⬢
    Application service orchestrates through ports only — PredictionApplicationService depends exclusively on domain port interfaces. It has no direct knowledge of Kafka, JDBC, or any specific adapter implementation.
  • ⬢
    Adapters are swappable at runtime — Repository implementations can be swapped between JDBC and in-memory variants via Spring @Profile without modifying a single line in the domain or application layers — proving true pluggability.
  • ⬢
    Strategy Pattern as a domain port — PredictionStrategy is defined in the domain layer and has five application-layer implementations (Crypto, Forex, Gold, Stock, Momentum), each encapsulating instrument-specific logic without infrastructure leakage.
  • ⬢
    Ubiquitous language throughout — Domain terms (MarketPrice, Prediction, Direction) are used consistently across all layers — from Kafka messages through strategy interfaces to database tables — reinforcing a shared domain vocabulary.
  • ⬢
    Testability by design — The layered architecture enables pure unit tests for domain and application logic (MockK, no Spring context) alongside full integration tests with Testcontainers — without modifying business code. Test boundaries mirror architectural boundaries.

Ports & Adapters Wiring

🔵 Domain Ports (interfaces)

  • PriceRepository — store, history, latest
  • PredictionPort — send prediction to Kafka
  • PredictionStrategy — predict direction from history
  • SentPredictionRepository — dedup per symbol/minute

🟢 Application Layer

  • PredictionApplicationService — orchestrates price ingestion + prediction dispatch
  • CryptoStrategy → implements PredictionStrategy
  • ForexStrategy → implements PredictionStrategy
  • GoldStrategy → implements PredictionStrategy
  • StockStrategy → implements PredictionStrategy
  • MomentumStrategy → implements PredictionStrategy

🟠 Driving Adapters (inbound)

  • MarketPriceConsumer — Kafka listener, calls onPriceReceived()
  • PredictionScheduler — cron trigger, calls sendPredictions()

🟠 Driven Adapters (outbound)

  • PredictionProducer → implements PredictionPort
  • JdbcPriceRepository → implements PriceRepository
  • JdbcSentPredictionRepo → implements SentPredictionRepository
  • InMemoryPriceRepo → implements PriceRepository (in-memory profile)
  • InMemorySentPredRepo → implements SentPredictionRepository (in-memory profile)
